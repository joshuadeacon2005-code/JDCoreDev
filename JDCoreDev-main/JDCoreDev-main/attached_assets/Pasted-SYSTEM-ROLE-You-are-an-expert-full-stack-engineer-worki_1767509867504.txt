SYSTEM / ROLE
You are an expert full-stack engineer working inside the existing JD Core Replit project. Your task is to implement:
1) A clear, controlled admin UI flow for approving, denying, or rescheduling meetings and office days.
2) A professional client-facing interaction and messaging flow so clients understand requests are pending and subject to confirmation.

Preserve all existing functionality, permissions, and architecture. Implement end-to-end: UI, routes, services, templates, email copy, and activity logs. Keep scope to this prompt only.

---

PRIMARY OBJECTIVE

A) ADMIN FLOW
- Admin must be able to:
  - Approve meeting requests
  - Deny meeting requests
  - Propose alternate times (same day or next day)
  - Approve / deny office day requests
- Admin must see clear status, context, and consequences before acting.

B) CLIENT FLOW
- Clients must clearly understand:
  - Meetings and office days are REQUESTS, not bookings
  - Requests may be denied or rescheduled
  - They can accept proposed alternatives easily
- Messaging must be calm, professional, and firm.

---

ADMIN UI FLOW (REQUIRED)

ADMIN PAGES TO UPDATE / ADD

1) Admin Inquiry Detail Page
Route:
- GET /admin/inquiries/{id}

Add a dedicated section: "Meeting Request"

Display:
- Requested date & time
- Meeting type (call / video)
- Duration
- Current meeting_status:
  - requested
  - proposed
  - confirmed
  - denied
  - cancelled

Admin Actions (buttons):
- [Confirm Meeting]
- [Deny Request]
- [Propose Alternate Time]

Confirm Meeting:
- Re-validate availability (no confirmed conflicts, lead time, office day rule).
- Set meeting status = confirmed.
- Block the calendar.
- Schedule reminders.
- Log activity_event: meeting_confirmed.

Deny Request:
- Require optional admin_notes (textarea).
- Set meeting status = denied.
- Cancel any pending reminders.
- Notify client via email.
- Log activity_event: meeting_denied.

Propose Alternate Time:
- Admin selects:
  - Same day OR next day
  - 1–3 alternate slots (MVP: max 2)
- Validate each proposed slot for availability.
- Create proposals and set meeting_status = proposed.
- Notify client with proposal options.
- Log activity_event: meeting_proposed.

---

2) Admin Office Day Requests Page
Route:
- GET /admin/office-days

For each request show:
- Client
- Project
- Date
- Status (requested / approved / denied)

Actions:
- Approve:
  - Status = approved
  - Schedule reminders
  - Log activity_event: office_day_approved
- Deny:
  - Status = denied
  - Cancel reminders
  - Log activity_event: office_day_denied

CLIENT-FACING FLOW (REQUIRED)

PUBLIC / CLIENT PAGES

1) Inquiry Submission Confirmation
After POST /inquiry:

Show message:
"Thanks for your inquiry.
If you requested a call, your request is pending confirmation.
You’ll receive an email once it’s confirmed or if another time is suggested."

No calendar blocking happens here.

---

2) Client Meeting Proposal Page
Route:
- GET /inquiry/meeting/{secure_token}

Display:
- Friendly explanation:
  "The originally requested time isn’t available.
Please choose one of the suggested alternatives below."

List proposed time options:
- Date
- Time
- Duration
- Meeting type

Actions:
- [Accept this time]
- [Decline all]

Accept:
- Re-validate availability.
- Set meeting status = confirmed.
- Cancel other proposals.
- Block calendar.
- Schedule reminders.
- Log activity_event: meeting_confirmed_from_proposal.

Decline all:
- Set meeting status = cancelled.
- Notify admin.
- Log activity_event: meeting_declined_by_client.

STATUS RULES (STRICT)

- requested:
  - Does NOT block calendar
- proposed:
  - Does NOT block calendar
- confirmed:
  - DOES block calendar
  - Schedules reminders
- denied / cancelled:
  - Never blocks calendar
  - Cancels reminders

---

- meeting confirmed
- office day approved

Create reminders for:

Channels:

Cancel reminders automatically if:
- meeting denied
