PART B — DATE “EXPECTED DATE RECEIVED STRING” AUDIT + FIX
===========================================================

PROBLEM STATEMENT
When creating/editing records with dates, app errors with messages like:
- “expected date received string”
This typically occurs when:
- HTML form inputs return strings (e.g., "2026-01-06")
- Pydantic/SQLAlchemy expects date/datetime objects
- Conversion/parsing is missing or inconsistent

AUDIT SCOPE (DATES)
Search repo for:
- Pydantic schemas with date/datetime fields
- Form handlers reading request.form or request data
- Fields: start_date, end_date, due_date, first_contact_date, expiry_date, decided_at, created_at, etc.
- Any custom validators or parsing helpers
- Any places calling model constructors directly using raw form strings

ROOT CAUSES TO CHECK (DO NOT SKIP)
- Using request.form strings directly to populate SQLAlchemy date columns
- Pydantic model expecting `date` but receiving `str` due to form submissions
- Mixed usage: some endpoints accept JSON (Pydantic parsing works) while others use form posts (needs manual parsing)
- Incorrect input formats (MM/DD/YYYY vs YYYY-MM-DD)
- Empty strings ("") passed for optional dates should become None (not parsed)
- Timezone-naive datetime strings handled incorrectly
- Using Jinja to render dates that aren’t serialized properly back into <input type="date"> values

REQUIRED FIX STRATEGY (DATES)
Implement a single shared parsing layer for date + datetime from form inputs, and use it in ALL form POST handlers.

1) Create utility module: app/utils/dates.py
- parse_date(value: str|None) -> date|None
  - Accepts:
    - None
    - "" => None
    - "YYYY-MM-DD" => date
  - Raises a clear ValueError for invalid formats.
- parse_datetime(value: str|None, assume_tz: str|None) -> datetime|None
  - For MVP accept:
    - ISO format strings from <input type="datetime-local"> like "YYYY-MM-DDTHH:MM"
  - Convert to UTC consistently OR store naive consistently (choose one and document).
- format_date_for_input(date|None) -> str
  - Returns "YYYY-MM-DD" or "" for <input type="date">

2) Normalize all POST handlers that create/update date fields:
For every form submission endpoint, BEFORE validation or DB write:
- Convert all date fields using parse_date
- Convert all datetime fields using parse_datetime
- Convert optional empty strings to None
Do not pass raw strings into Pydantic/SQLAlchemy expecting date objects.

3) Pydantic schema adjustments (if used for forms)
If you use Pydantic for form handling:
- Either:
  a) Switch date fields to `date` but ensure you pass parsed date objects
  OR
  b) Allow strings with validators:
     - @field_validator(..., mode="before") to parse date strings
Pick one approach consistently.

4) Improve error reporting (developer-friendly)
When parsing fails:
- Return a form error message that identifies which field is invalid and what format is expected:
  e.g. “Due date must be YYYY-MM-DD”
- Log the original value and route for debugging (server logs).

5) Fix templates for date inputs
Ensure date inputs use format_date_for_input so editing existing records works:
- <input type="date" value="{{ some_date|date_input }}">
Register a Jinja filter `date_input` that uses format_date_for_input.

6) Fix seed data
Ensure seed uses actual date objects (datetime.date) or valid ISO strings parsed through the same helper.

7) Tests (minimum for dates)
- parse_date handles None, "", valid "YYYY-MM-DD"
- parse_date rejects invalid formats
- Create/update milestone with due_date via form post succeeds (no “expected date received string”)
- Create/update client with first_contact_date succeeds
